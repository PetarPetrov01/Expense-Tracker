# Expense Tracker v2 — Feature Brainstorm

_Date: 2026-05-17_
_Status: Brainstorm (pre-spec). Awaiting user direction before promoting to a Phase 2 plan._

## TL;DR

- **Headline feature:** structured **JSON import/export with schema versioning, dry-run preview, and deterministic dedupe by content-hash** — solves the user's stated ask, doubles as free-forever disaster recovery, and is the natural foundation for every later "data hygiene" feature.
- **Top constraint:** every Phase 2 idea must pass two gates — **offline-safe** and **free-forever**. That kills cloud sync, OCR APIs, FX rate APIs, and anything requiring an account. It does not kill recurring expenses, budgets, search, biometric lock, light mode, or persisted settings.
- **Recommended Phase 2 size:** ship **5 features** in one release — Import/Export, Persisted Settings (incl. dark/light + locale + currency), Search & Filter, Budgets, and Recurring Expenses. Total effort estimate: ~M+ (a focused 2–3 week solo slot). Everything else parks in Phase 3.

---

## The headline ask: Import / Export

User quote:
> "One feature I am thinking is something like an export — so I can load up my expenses from a year ago for example (like a specific format it would accept and run entries against the database)."

Read carefully: this is **round-trip data interchange**, not just a one-way dump. It needs to be re-importable into a future install (lost phone, factory reset, new device) and ideally human-inspectable (so the user trusts what's in their backup).

### Format choice — recommendation: JSON (primary) + raw SQLite copy (escape hatch)

| Format | Pros | Cons | Verdict |
|---|---|---|---|
| **CSV** | Human-readable in any spreadsheet; tiny | No schema, no nested types, no version metadata, no categories table (would need 2 CSVs or denormalised join). Notes with commas/newlines are a parsing minefield. | **No** as primary. Maybe a future "export to spreadsheet" view-only side-feature. |
| **JSON** | Self-describing, versionable, carries categories + expenses + metadata in one doc, trivial to validate with `zod` (already a dep), trivial to diff, easy to hand-edit. | Slightly bigger files. Not opens-in-Excel. | **Yes — primary format.** |
| **SQLite file copy** | _Is_ the ground truth. Perfect fidelity, zero conversion bugs. | Opaque binary blob; can't be inspected without tools; couples backups to the exact schema version — restoring an old `.db` into a newer app means running migrations on import. | **Yes — secondary, as a "raw backup" escape hatch.** Useful for "I just want a perfect copy before I wipe my phone." Not for selective import. |
| **All three** | Maximum flexibility | Triples surface area for v2. CSV adds the most cost for the least benefit. | **No.** |

**Recommendation:** Ship **JSON as the canonical interchange format** and offer a **"Copy database file"** action as a second button for paranoid backups. Skip CSV in v2.

### JSON shape (proposed)

```json
{
  "format": "expense-tracker-export",
  "formatVersion": 1,
  "appVersion": "2.0.0",
  "exportedAt": "2026-05-17T12:34:56.000Z",
  "currency": "EUR",
  "categories": [
    {
      "stableId": "seed:groceries",
      "name": "Groceries",
      "icon": "cart",
      "color": "#4CAF50",
      "isSeed": true,
      "createdAt": "2026-01-01T00:00:00.000Z"
    },
    {
      "stableId": "user:c5f1...e2",
      "name": "Cat cafés",
      "icon": "cat",
      "color": "#FF9800",
      "isSeed": false,
      "createdAt": "2026-03-04T18:11:00.000Z"
    }
  ],
  "expenses": [
    {
      "contentHash": "sha1:9b...",
      "amountCents": 1234,
      "categoryStableId": "seed:groceries",
      "note": "Lidl, late shop",
      "occurredAt": "2026-05-17T19:02:00.000Z",
      "createdAt": "2026-05-17T19:05:11.000Z"
    }
  ]
}
```

Key design choices:

- **`stableId` on categories**, not numeric `id`. Numeric DB ids are local and meaningless across installs. Seed categories use a deterministic key (`seed:groceries`); user-created ones get a generated random suffix on first export and stick. Stored as a new nullable column `stableId` on the `categories` table (filled lazily on first export, present from creation in v2+).
- **`contentHash` on expenses** — a SHA-1 of `(amountCents | occurredAt-iso | categoryStableId | normalisedNote)`. Computed at export time, recomputed at import time. This is the **dedupe key**. Storing it inside the JSON makes dry-run preview cheap (single hash scan vs N row comparisons).
- **`formatVersion: 1`** at the top — see versioning below.
- ISO 8601 strings, not epoch ms, in the JSON. The DB stays epoch-ms. Conversion is one helper.

### Schema versioning — what happens when v3 adds a column?

Rule: **import is permissive across minor changes, strict across major ones.**

- **`formatVersion`** in the file declares what shape it claims to be.
- Importer carries a **migration chain**: `migrateFormatV1toV2`, `migrateFormatV2toV3`, ... Each is a pure function `oldDoc -> newDoc`. Apply in sequence until the in-memory doc matches the current `formatVersion`. Then validate against a `zod` schema for the current version. Then insert.
- **Older app, newer file:** if `file.formatVersion > app.formatVersion`, refuse with a clear message ("This backup was made by a newer version of the app. Update the app and try again.") rather than guessing.
- **New optional columns:** add them as optional in the current `zod` schema. v1 files just won't have them; they get default values.
- **Required new columns:** force a migration step that fills them. Example: v3 adds `currency` per expense — migration sets it to the file-level `currency` field for every v1/v2 row.
- **Removed columns:** migration drops them silently.
- **Renamed columns:** migration renames in place.

This keeps the importer code linear and inspectable, and the user never sees "import failed, unknown field."

### Conflict resolution — recommendation: hash-based skip with override

Compute `contentHash` for every expense in the file AND for every existing local expense (cheap: index it once on import). Then:

| Local state | File state | Action |
|---|---|---|
| Hash exists locally | Hash in file | **Skip** (true duplicate) |
| Hash not in local | Hash in file | **Insert** |
| Hash exists locally | Hash NOT in file | Leave alone (no deletion from import) |

That's the default. Also offer **two user-selectable modes** on the import screen:

1. **Merge (recommended, default):** the table above. Idempotent — re-importing the same file is a no-op.
2. **Replace all:** wipes local expenses + categories before importing. Useful for "restore from backup after factory reset." Confirmed by a typed "REPLACE" string to avoid catastrophe.

Explicitly **do not** offer "always insert" (creates duplicates) or "ask per row" (UX disaster at 1,000 rows). Hash-based merge handles 99% of cases correctly.

### Category resolution on import

Order of resolution for each `categoryStableId` referenced by an expense in the file:

1. **Match by `stableId`** on existing local categories. If found, use it.
2. **Match by `name`** (case-insensitive, trimmed) on existing local categories. If found, **adopt the file's `stableId` onto the local row** so future round-trips align. Use it.
3. **Create it.** Take name/icon/color from the file's `categories` block. Mark `isSeed: false` regardless of what the file said — seed status is an app-install concept, not a backup concept. Use it.
4. (Edge) Expense references a `categoryStableId` not listed in the file's `categories` array → **fall back to a synthetic "Imported (unknown)" category**, created once with a distinctive grey color. Surface a count in the import summary.

No "ask the user per category" flow. The above resolves every case automatically and the dry-run preview shows what will happen.

### Where the file lives on Android

Use the **Storage Access Framework (SAF)** via Expo libraries:

- **Export:** `expo-file-system` writes to app cache, then `expo-sharing.shareAsync(...)` invokes the system share sheet → user picks Drive / Files / Telegram-to-self / wherever. No bespoke folder permissions needed. New deps: `expo-sharing`.
- **Import:** `expo-document-picker.getDocumentAsync({ type: 'application/json' })` returns a content:// URI; read it with `expo-file-system.readAsStringAsync`. New deps: `expo-document-picker`.
- **Raw DB copy export:** read `${FileSystem.documentDirectory}SQLite/expense-tracker.db` as base64 or stream into the share sheet. Same library set.

Both libs are first-party Expo, ship via dev-client rebuild (we already build a signed APK), free forever, fully offline. No extra Android manifest fiddling — SAF goes through the system picker.

### UX

A new **Settings → Data** subscreen with four actions, in this order:

1. **Export to JSON** — generates file, opens share sheet. Toast on success.
2. **Import from JSON** — opens document picker → reads file → validates → shows **Dry-run Preview** (modal): "This will add 47 expenses, skip 12 duplicates, create 2 new categories: 'Cat cafés', 'Dentist'. Continue?" with Confirm / Cancel. Only on Confirm does anything touch the DB. The dry-run is non-negotiable for trust.
3. **Replace from JSON** (destructive, separate button, red) — same flow but the confirmation requires typing "REPLACE".
4. **Copy raw database file** — emergency backup escape hatch.

No buttons on the home screen, no FAB collision, no notification. This is settings-tier infrastructure, not daily UX.

Progress feedback: for files under ~1MB (well above realistic personal use) the whole import runs in <1s on-device, so a spinner overlay is enough. No progress bar needed. Wrap the actual DB writes in a single Drizzle transaction so partial failures roll back cleanly.

### Recommendation summary

| Question | Recommendation |
|---|---|
| Format | JSON primary, raw SQLite file as secondary escape hatch. No CSV in v2. |
| Versioning | `formatVersion` field + linear migration chain + zod validation. Reject newer-than-app files. |
| Dedupe | SHA-1 content hash of (amount, occurredAt, categoryStableId, normalisedNote). Default action: skip duplicates. |
| Missing categories | Auto-resolve by stableId → name → create from file. Synthetic fallback for truly unknown. |
| Storage | SAF via `expo-document-picker` + `expo-sharing` + `expo-file-system`. |
| UX gate | Mandatory dry-run preview before any write. Transactional writes. |

## Other Features Considered

15 ideas, ranked roughly within each tier but presented unranked here. Effort is calibrated to a single solo developer working evenings: **S** ≈ a day, **M** ≈ a few days, **L** ≈ a week or more.

### 1. Persisted Settings (currency, theme, locale)

- **Desc:** Move Zustand-only settings into a `settings` SQLite table (or AsyncStorage). Currency, theme, locale, default category survive app restart.
- **Value:** High annoyance fix. Currency resetting to EUR every cold start is a paper cut on every launch.
- **Cost:** S. One small table or AsyncStorage namespace; hydrate on app boot; tiny `useSettings` Zustand store with a write-through to disk.
- **Offline-safe?** Yes.
- **Free-forever?** Yes.
- **Risk:** Race condition on first launch (UI renders before hydrate completes). Mitigation: gate root layout on a `settingsLoaded` flag, show splash until true.

### 2. Light Mode + System Theme Follow

- **Desc:** Add a second palette to `src/theme.ts`, expose `theme: 'system' | 'dark' | 'light'` setting.
- **Value:** Daytime outdoor readability. Cheap win.
- **Cost:** S. Refactor theme to a context provider; existing screens already pull colors from theme — minimal touch.
- **Offline-safe?** Yes.
- **Free-forever?** Yes.
- **Risk:** Hardcoded hex values scattered in components. Audit needed. Cost stays S if `theme.ts` is well-respected today; bumps to M if not.

### 3. Locale-aware Number Formatting

- **Desc:** Replace hardcoded `€12.34` with `Intl.NumberFormat`. User picks locale or "follow device."
- **Value:** Real for European users — `12,34 €` vs `€12.34` matters cognitively.
- **Cost:** S. `Intl.NumberFormat` is in Hermes. One helper, one setting.
- **Offline-safe?** Yes.
- **Free-forever?** Yes.
- **Risk:** Bundle size of full ICU data on Android. Hermes ships with `Intl` — verify before depending on it. Fallback: ship a tiny custom formatter.

### 4. Search & Filter

- **Desc:** Search expenses by note text, filter by category, filter by date range. Replace the "recent 50" home list with a searchable list screen.
- **Value:** The user's headline ask ("load up my expenses from a year ago") is partly a _read_ ask, not just import/export. Search closes the loop.
- **Cost:** M. New screen + query layer. Drizzle handles it. Debounced text input, multi-select category chips, range pickers reusing existing `datetimepicker`.
- **Offline-safe?** Yes.
- **Free-forever?** Yes.
- **Risk:** Performance at 5k+ rows. SQLite handles it fine but unindexed `LIKE` on `note` is the slow path. Add an index; consider FTS5 only if it gets slow.

### 5. Budgets / Monthly Caps

- **Desc:** New `budgets` table: `(id, categoryId nullable, amountCents, period: 'monthly')`. Null categoryId = overall budget. Home screen shows progress bars; over-budget categories turn red.
- **Value:** The thing that turns a logger into a tool. Personal-finance core.
- **Cost:** M. New table, new screen (Settings → Budgets), home-screen widget, threshold logic.
- **Offline-safe?** Yes.
- **Free-forever?** Yes.
- **Risk:** Scope creep — weekly/yearly/rollover/per-day variants. **Cap v2 at monthly only.** Resist.

### 6. Recurring Expenses

- **Desc:** New `recurring_expenses` table with rule (`monthly_on_day=1`, `weekly_on_dow=5`, etc.) and template (amount, category, note). On app open, materialise any rules whose next occurrence has passed into real `expenses` rows.
- **Value:** Rent, subscriptions, salary tracking. Removes friction from the most predictable expenses.
- **Cost:** M. Schema + scheduler-on-open + UI to manage rules + visible "auto-added on 2026-05-01" badge on materialised rows.
- **Offline-safe?** Yes — explicitly avoid AlarmManager / background tasks. Materialise on app open only. Simple, robust, no battery footprint.
- **Free-forever?** Yes.
- **Risk:** Double-materialisation if the scheduler logic forgets to advance `lastMaterialisedAt`. Idempotency by tracking last run per rule. Also: app not opened for 3 months → 3 rent rows appear on next open. Acceptable; arguably correct.

### 7. Quick-Add via Long-Press FAB

- **Desc:** Long-press the home FAB → instant-add an expense with last-used category and today's date, prompting only for amount.
- **Value:** Reduces add-expense from 3+ taps to ~2 for the common case.
- **Cost:** S. Reuse existing add modal in a "minimal" mode. Track `lastUsedCategoryId` (already wants to be persisted — see #1).
- **Offline-safe?** Yes.
- **Free-forever?** Yes.
- **Risk:** Discoverability — long-press is invisible. Mitigate with a one-time tooltip on first home-screen view after update.

### 8. Receipt Photo Attachment

- **Desc:** Optional image attached to an expense via `expo-image-picker`. Stored in app document directory, path saved in new `expenses.receiptPath` column.
- **Value:** Useful for reimbursable/warranty receipts. Real-world handy.
- **Cost:** M. New dep, new column, storage management (orphan cleanup on delete), thumbnail in list view, full view on tap.
- **Offline-safe?** Yes (local file).
- **Free-forever?** Yes.
- **Risk:** Storage growth. Mitigate with on-device image compression (long-edge 1600px, JPEG 75) before save. Also: export grows. **Decision: receipts are NOT included in JSON export in v2.** They live only in the raw DB copy. (Adding them to JSON means base64-bloating the export 50×.) Document this trade-off.

### 9. Statistics Depth (year-over-year, top categories, biggest expense, weekday averages)

- **Desc:** Add 4–5 new cards to the existing Stats screen.
- **Value:** Modest. Nice-to-have. The current stats already cover 80% of use.
- **Cost:** M. Pure query work; chart components already exist.
- **Offline-safe?** Yes.
- **Free-forever?** Yes.
- **Risk:** Visual clutter on a phone-sized screen. Needs design discipline — collapse-by-default cards, or a "More stats" subscreen.

### 10. Biometric App Lock

- **Desc:** `expo-local-authentication` gate on app open. Setting: off / on-cold-start / on-every-resume.
- **Value:** Real for shared-device or pickpocket scenarios. Personal financial data.
- **Cost:** S. One library, one boolean setting, one gate screen.
- **Offline-safe?** Yes.
- **Free-forever?** Yes.
- **Risk:** Lockout if biometrics are removed/reset on the device. Always offer a "skip" with the device PIN fallback that `expo-local-authentication` already provides.

### 11. Auto-Backup to Phone Folder

- **Desc:** Schedule a recurring on-app-open check: if N days since last auto-export, write a JSON snapshot to a user-chosen SAF folder. Keep last K (e.g. 8). Setting: off / weekly / monthly.
- **Value:** Disaster recovery without cloud. Combined with #4 (search) and Import, this is "I lost my phone and got everything back."
- **Cost:** S+ (assuming import/export already exists). Just a scheduler + retention loop + SAF folder picker.
- **Offline-safe?** Yes — writes to local storage / user's chosen folder.
- **Free-forever?** Yes.
- **Risk:** SAF persisted folder URIs can go stale (folder deleted/moved). Detect, warn, prompt to re-pick.

### 12. Multi-Currency Entries

- **Desc:** Each expense stores its native currency + amount. Stats convert to display currency using a **manually-entered rates table** (no API).
- **Value:** Travellers. Niche but real.
- **Cost:** L. Schema change, FX table CRUD, conversion-aware queries, UI in add/edit screen, stats re-think.
- **Offline-safe?** Yes only if FX rates are manual.
- **Free-forever?** Yes only if FX rates are manual.
- **Risk:** Manual FX rates are tedious and stale. Auto-fetching crosses the "no required network calls" line. **Verdict: not worth v2.** Park in Phase 3 and reconsider when there's a real travel use-case.

### 13. CSV Export (one-way, view-only)

- **Desc:** Separate from the JSON round-trip: a one-way CSV dump for spreadsheet analysis.
- **Value:** Minor — for users who want to pivot in Excel.
- **Cost:** S, _but_ only after JSON export exists.
- **Offline-safe?** Yes.
- **Free-forever?** Yes.
- **Risk:** Low. Mostly: scope-creep risk during the main import/export build. Add later if asked.

### 14. Android Home-Screen Widget (quick-add)

- **Desc:** Tap a widget → opens app directly to add-expense modal.
- **Value:** Real friction reduction. Loved by hardcore loggers.
- **Cost:** L. Android widgets in Expo require either a custom native module (config plugin) or one of the half-maintained community libs. Significant lift, fragile across Android versions.
- **Offline-safe?** Yes.
- **Free-forever?** Yes (no cloud), but high _maintenance_ cost.
- **Risk:** The build complexity alone could derail Phase 2. **Skip.** Revisit only if Expo ships first-party widget support.

### 15. Tags / Multi-Category per Expense

- **Desc:** A second classification axis. E.g. category=Food, tags=[work, travel].
- **Value:** Marginal for a single user. Categories already do the heavy lifting.
- **Cost:** M. New `tags` + `expense_tags` join table, UI, stats re-think.
- **Offline-safe?** Yes.
- **Free-forever?** Yes.
- **Risk:** Personal-use apps that add tags rarely see them used. **Skip** unless the user specifically asks. Note: budgets (#5) are a better answer to "I want to slice my data more ways."

## Phase 2 Plan Outline

Build order matters here because there are real dependencies and the smart sequencing front-loads risk.

```
[1] Persisted Settings  ──┐
                          ├──► [2] Import / Export (JSON + DB copy)
                          │
                          ├──► [3] Search & Filter
                          │
                          ├──► [4] Budgets (monthly)
                          │
                          └──► [5] Recurring Expenses
```

**Why this order:**

1. **Persisted Settings first.** Everything else benefits. Currency stickiness is a precondition for correct export (`currency` field in JSON should reflect a stable value, not "whatever was in memory at export time"). It's also small enough to be the warm-up.
   - Soft-include: theme + locale formatting if the theme refactor turns out to be cheap. Decide at start of the slot.

2. **Import / Export second.** The headline feature, done while the codebase is still small enough that the schema is easy to reason about and the migration chain starts at v1 instead of v3. Doing this _before_ Budgets and Recurring means we don't have to retro-fit those new tables into the export format mid-stream.
   - Sub-steps: stableId migration on categories → contentHash helper → zod schema for formatVersion 1 → export pipeline → import pipeline → dry-run preview UI → SAF wiring → raw DB copy → Replace-all flow.

3. **Search & Filter third.** Independent of #4 and #5. Pure read-side feature. Closes the user's stated "load up expenses from a year ago" loop.

4. **Budgets fourth.** Schema addition. Now the export format version bumps to v2 (adds optional `budgets` block); we exercise the migration chain we just built. Good integration test of #2.

5. **Recurring Expenses fifth.** Same story — exercises the format-versioning machinery again (v2 → v3 adds `recurringRules` block), and the on-app-open materialiser is conceptually the heaviest piece, best done when everything else is stable.

**Cross-cutting tasks (sprinkled throughout, not their own slot):**

- One DB migration per new table (Drizzle Kit generates).
- `stableId` backfill migration for existing seed + user categories.
- Update the Stats screen if Budgets affects it (likely a "vs budget" overlay on the monthly chart — small).

**Estimated calendar:** 2–3 weeks of evening work, gated on no scope creep in Budgets and Recurring.

## Open Questions for the User

These are things a normal brainstorming session would surface as questions. The user asked me not to block on them, so each is paired with the default I chose and the reasoning.

1. **Q: Should the JSON export include `categories` even when no user-created ones exist?**
   **Default:** Yes, always. Including seed categories makes the export self-contained and survives the case where the importing app version has different seed defaults.
   **Why this default is safe:** larger file by ~1KB, much more robust. Easy to revisit.

2. **Q: Should `Replace from JSON` also replace categories, or only expenses?**
   **Default:** Replaces both. "Replace all" should mean exactly that — a clean slate restore-from-backup. Partial replace is confusing.
   **Risk if wrong:** A user wanting to keep their custom category list while wiping expenses has no path. Mitigated by the dry-run preview clearly stating "will delete all categories and expenses."

3. **Q: Budgets — does "over budget" trigger a notification or just visual feedback?**
   **Default:** Visual only in v2 (red bar, red home-screen number). Notifications require `expo-notifications` + permission flow + an on-open re-check; that's a 3-day rabbit hole disguised as a 1-hour feature.
   **Revisit:** Add notifications in Phase 3 if the user asks.

4. **Q: Recurring expenses — what date does the materialised row carry as `occurredAt`?**
   **Default:** The _scheduled_ date (e.g., the 1st of the month), not the date the user happened to open the app. Critical for stats accuracy. `createdAt` reflects when the app actually inserted it.
   **Risk if wrong:** Stats look weird (rent appears on the 17th instead of the 1st). Mitigated by being explicit in the materialiser logic and unit-testing it.

5. **Q: Should Search & Filter live on the Home tab or as its own tab?**
   **Default:** Tap on "Recent expenses" header on Home navigates to a full-screen searchable list. No new tab — the tab bar is already at the right density.
   **Risk if wrong:** Discoverability. A "View all" link on the recent-list section solves it.

6. **Q: Light mode — should it ship in Phase 2?**
   **Default:** Soft yes, _if_ the theme refactor turns out to be 1 day. The audit pass at the start of Phase 2 decides. Treat it as a known-low-cost option to pull in if the schedule allows.
   **Riskiest open question** — the cost ranges from "trivial" to "M" depending on how disciplined v1 components were about using `theme.ts` rather than inline hex values. Flagged for first-day audit.

7. **Q: Currency persistence — switch to AsyncStorage or to a new `settings` SQLite table?**
   **Default:** New `settings` SQLite table (`key TEXT PRIMARY KEY, value TEXT`). Reasons: (a) backups already cover SQLite, so settings get backed up too; (b) consistent with the rest of the data layer; (c) AsyncStorage on Android is a separate file/store the user can't see in their DB copy.
   **Trade-off:** Slightly more boilerplate than AsyncStorage. Worth it.
